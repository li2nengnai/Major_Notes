# [[异常处理]]

## 核心概念

### 异常是什么

程序运行中出现的不正常情况就叫异常。Java 把所有异常分成了三类：

```
Throwable（所有问题）
├── Error（严重错误，程序无法处理）
│   ├── OutOfMemoryError           ← 内存溢出（堆满了）
│   ├── StackOverflowError         ← 栈溢出（递归太深）
│   └── NoClassDefFoundError       ← 类找不到
│
└── Exception（可以处理）
    ├── 受检异常（Checked，编译器强制你处理）
    │   ├── IOException            ← 文件读写失败
    │   ├── SQLException           ← 数据库操作失败
    │   ├── ParseException         ← 日期解析失败
    │   └── ClassNotFoundException ← 类没找到
    │
    └── 运行时异常（RuntimeException，编译器不强制）
        ├── NullPointerException       ← 空指针（最常见！）
        ├── ArrayIndexOutOfBoundsException ← 数组越界
        ├── IllegalArgumentException   ← 非法参数
        ├── ClassCastException         ← 类型转换失败
        └── ArithmeticException        ← 算术异常（除0）
```

**核心区别**：受检异常不处理→编译**不通过**；运行时异常不处理→编译通过，跑起来才崩。

### `throws` vs `throw`（面试必问）

| 关键字 | 位置 | 作用 | 后跟 |
|--------|------|------|------|
| `throws` | 方法声明上 | 声明"我这个方法可能抛异常，调用者注意" | 异常类型 |
| `throw` | 方法体内 | 手动抛出一个异常对象 | 异常对象 |

```java
public void saveAge(int age) throws AgeIllegalException {
    if (age < 0 || age > 150) {
        throw new AgeIllegalException("年龄不合法");
    }
}
```

**记忆口诀**：`throw` 抛（动作），`throws` 声明（态度）。

### 异常处理三种方式

```java
// 方式1：try-catch 自己处理
try {
    int r = 10 / 0;
} catch (ArithmeticException e) {
    System.err.println("除数不能为0：" + e.getMessage());
} catch (Exception e) {
    e.printStackTrace();  // 打印完整堆栈
} finally {
    System.out.println("无论是否异常都会执行");
}

// 方式2：throws 抛给上层处理
public void readFile(String path) throws IOException {
    Files.readAllBytes(Paths.get(path));
}

// 方式3：try-with-resources（JDK7+，自动关资源）
try (FileInputStream fis = new FileInputStream("test.txt");
     BufferedReader br = new BufferedReader(new InputStreamReader(fis))) {
    String line = br.readLine();
    System.out.println(line);
} catch (IOException e) {
    e.printStackTrace();
}  // 不用写 finally 关资源，自动关
```

### 自定义异常

```java
// 编译时异常 → 继承 Exception
public class AgeIllegalException extends Exception {
    private int code;
    public AgeIllegalException(String msg) {
        super(msg);
        this.code = 400;
    }
}

// 运行时异常 → 继承 RuntimeException（常用，不用写 throws）
public class BusinessException extends RuntimeException {
    private int code;
    public BusinessException(int code, String msg) {
        super(msg);
        this.code = code;
    }
}

// 使用
public void checkAge(int age) {
    if (age < 0) {
        throw new BusinessException(400, "年龄不能为负数");
    }
}
```

### 异常最佳实践

```java
// ✅ 1. 精确捕获，不要吞异常
try { /* ... */ } catch (SpecificException e) {
    log.error("操作失败", e);
    throw e;  // 或包装后重新抛出
}

// ✅ 2. 异常日志要打完整堆栈（e.getMessage() 不够，要 e.printStackTrace() 或 log.error(..., e)）
log.error("处理失败，orderId={}", orderId, e);  // 带上上下文参数

// ✅ 3. 资源始终用 try-with-resources
try (Connection conn = getConnection()) { /* ... */ }

// ❌ 4. 不要用异常控制正常流程（性能差）
try {
    Integer.parseInt(str);
} catch (NumberFormatException e) {
    // 正常逻辑放 catch 里？❌ 不要！
}
// 应改为：if (str.matches("\\d+")) { ... }
```

### 异常性能

创建异常对象需要填充栈轨迹（`fillInStackTrace()`），开销大。**不要用异常控制正常流程**。

## 踩坑记录

> [!warning] finally 中写 return
> **现象**：try 里 return 1，结果方法返回了 2
> ```java
> public static int test() {
>     try { return 1; }
>     finally { return 2; }  // ❌ finally 的 return 覆盖了 try 的
> }
> // test() → 2
> ```
> **原则**：finally 只用来释放资源，不要返回数据。

> [!warning] 空 catch 块吞异常
> ```java
> try { /* 可能抛异常 */
> } catch (Exception e) { }  // ❌ 线上出问题了连日志都没有
> ```
> **治标**：至少打日志
> **治本**：封装统一异常处理

> [!warning] 异常捕获太宽泛
> ```java
> try { /* ... */ }
> catch (Exception e) { }  // ⚠️ 把 NPE 和 IOException 一起抓了，没法区分处理
> ```
> **建议**：精确捕获具体异常类型，需要多个分开处理。

> [!warning] 受检异常忘记处理
> **现象**：编译报错 `Unhandled exception: java.text.ParseException`
> **原因**：`SimpleDateFormat.parse()` 声明了 throws ParseException，调用者必须处理。
> **解决**：try-catch 或方法签名上加 throws。

## 拓展延伸

### Spring 事务中的异常

默认只回滚 `RuntimeException` 和 `Error`，受检异常不回滚。需要 `@Transactional(rollbackFor = Exception.class)`。

### 异常 vs 错误码

| 方式 | 优点 | 缺点 |
|------|------|------|
| 异常 | 强制处理、堆栈信息完整 | 性能开销 |
| 错误码 | 性能好 | 容易忽略返回值 |
