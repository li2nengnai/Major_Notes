# [[并发编程]]核心知识

## 核心概念

### 线程 vs 进程

| 对比 | 进程 | 线程 |
|------|------|------|
| 资源 | 独立内存空间 | 共享进程资源 |
| 切换开销 | 大 | 小 |
| 通信方式 | IPC（管道/消息队列） | 共享内存 |

**线程状态**：
```
NEW → RUNNABLE → BLOCKED/WAITING/TIMED_WAITING → TERMINATED
```

### 创建线程三种方式

```java
// 1. 继承 Thread（简单但 Java 单继承有局限性）
class MyThread extends Thread {
    public void run() { /* 业务代码 */ }
}
new MyThread().start();

// 2. 实现 Runnable（推荐，不影响继承其他类）
class MyTask implements Runnable {
    public void run() { /* 业务代码 */ }
}
new Thread(new MyTask()).start();

// 3. Callable + FutureTask（带返回值，能抛异常）
FutureTask<Integer> ft = new FutureTask<>(() -> {
    return 42;  // 有返回结果
});
new Thread(ft).start();
Integer result = ft.get();  // 阻塞等待结果
```

### synchronized 原理与锁升级

**三种用法**：
```java
// 1. 同步代码块（锁对象）
synchronized (this) { /* 线程安全操作 */ }

// 2. 同步方法（锁当前对象）
public synchronized void method() { }

// 3. 静态同步方法（锁 Class 对象）
public static synchronized void staticMethod() { }
```

**锁升级（JDK6 优化，面试必问）**：
```
无锁 → 偏向锁（单线程） → 轻量级锁（CAS自旋） → 重量级锁（OS互斥）
```

| 锁状态 | 特点 | 适用场景 |
|--------|------|---------|
| 偏向锁 | 锁对象头记录线程ID | 单线程反复获取同一把锁 |
| 轻量级锁 | CAS 自旋，不阻塞 | 少量线程交替执行 |
| 重量级锁 | 线程阻塞，OS 调度 | 多线程竞争激烈 |

**JDK6 锁优化**：锁粗化（相邻同步块合并）、锁消除（JIT 检测无竞争时删锁）、适应性自旋

### volatile 与 JMM

**volatile 两个语义**：
1. **可见性**：写操作立即刷新到主存，读操作从主存读（不读缓存）
2. **禁止指令重排序**：通过内存屏障实现（防止 `double-checked locking` 出问题）

```java
// volatile 不能保证原子性！
volatile int count = 0;
count++;  // 这行不是原子操作！= 读 count → 加 1 → 写回 count
// 三个步骤可能被线程切换打断
```

**JMM 三大特性**：

| 特性 | 说明 | 实现 |
|------|------|------|
| 原子性 | 操作不可中断 | `synchronized` / `Lock` / `Atomic` 类 |
| 可见性 | 修改对其他线程可见 | `volatile` / `synchronized` |
| 有序性 | 禁止指令重排 | `volatile` / `synchronized` |

### AQS（AbstractQueuedSynchronizer）

**AQS 是整个 JUC 锁的基石**（面试高频）。

**核心**：CLH 变体队列 + `volatile int state`

```
        ┌─────────────────────────────────┐
        │   AbstractQueuedSynchronizer    │
        │   state (volatile int)          │  ← 资源状态
        │   head → Node ⇄ Node ⇄ Node     │  ← 等待队列
        └─────────────────────────────────┘
```

**模板方法**：
- `tryAcquire()` / `tryRelease()` — 独占锁（如 ReentrantLock）
- `tryAcquireShared()` / `tryReleaseShared()` — 共享锁（如 Semaphore）

**应用**：ReentrantLock、CountDownLatch、Semaphore、ThreadPoolExecutor 全部基于 AQS

### 线程池

**七个核心参数**：

```java
ThreadPoolExecutor pool = new ThreadPoolExecutor(
    corePoolSize,           // 核心线程数（一直活着，除非 allowCoreThreadTimeOut）
    maximumPoolSize,        // 最大线程数
    keepAliveTime,          // 临时线程空闲存活时间
    TimeUnit.SECONDS,
    new LinkedBlockingQueue<>(queueSize),  // 任务队列
    Executors.defaultThreadFactory(),      // 线程工厂
    new ThreadPoolExecutor.AbortPolicy()   // 拒绝策略
);
```

**工作流程**：
```
提交任务
   ↓
核心线程未满？ → 是 → 创建核心线程执行
   ↓ 否
任务队列未满？ → 是 → 入队等待
   ↓ 否
最大线程未满？ → 是 → 创建临时线程执行
   ↓ 否
执行拒绝策略
```

**四种拒绝策略**：

| 策略 | 行为 |
|------|------|
| `AbortPolicy`（默认） | 抛 `RejectedExecutionException` |
| `CallerRunsPolicy` | 调用者线程自己执行（压回调用方） |
| `DiscardPolicy` | 直接丢弃，不报错 |
| `DiscardOldestPolicy` | 丢弃队列中最老的任务 |

**禁止 Executors 创建线程池**：
```java
Executors.newFixedThreadPool(10);    // ❌ LinkedBlockingQueue 无界可能 OOM
Executors.newCachedThreadPool();     // ❌ 最大线程 Integer.MAX_VALUE
// ✅ 手动 new ThreadPoolExecutor()，明确参数
```

### JUC 常用工具类

| 类 | 作用 | 场景 |
|------|------|------|
| `CountDownLatch` | 一组线程等待另一组线程完成 | 主线程等所有子线程执行完 |
| `CyclicBarrier` | 一组线程互相等待到齐 | 多线程同时开始某阶段 |
| `Semaphore` | 信号量，控制并发数量 | 限流（比如最多 3 个线程同时访问） |
| `ReentrantLock` | 可重入锁（比 synchronized 灵活） | 需要中断锁/超时锁/公平锁 |
| `ReentrantReadWriteLock` | 读写分离锁 | 读多写少场景 |
| `ThreadLocal` | 线程本地变量（每个线程一份副本） | 存放用户 Session/数据库连接 |
| `atomic` 包 | CAS 原子操作类 | `AtomicInteger` / `AtomicReference` |

## 踩坑记录

> [!warning] volatile 不保证原子性
> **现象**：多线程 `count++` 结果比预期小。
> **原因**：`count++` 是读-改-写三步操作，volatile 只能保证可见性，不能保证原子性。
> **解决**：用 `AtomicInteger` 或 `synchronized`。
> ```java
> AtomicInteger count = new AtomicInteger(0);
> count.incrementAndGet();  // CAS 原子自增
> ```

> [!warning] ThreadLocal 用完不 remove() 导致内存泄漏
> **原因**：ThreadLocalMap 的 key 是弱引用（GC 可能回收），但 value 是强引用。key 被回收后 value 永远无法访问。
> **解决**：用完调 `remove()`，尤其在**线程池**中（线程复用，不 remove 会让下个任务读到脏数据）。
> ```java
> ThreadLocal<User> tl = new ThreadLocal<>();
> try {
>     tl.set(user);
>     // ... 业务逻辑
> } finally {
>     tl.remove();  // 必须！
> }
> ```

> [!warning] 线程池核心线程数设多少
> **现象**：设了 200 个线程但 CPU 才 4 核，上下文切换成了性能瓶颈。
> **公式**：
> - CPU 密集型：`N + 1`（N = CPU 核数）
> - IO 密集型：`2N`（线程等 IO 时 CPU 可以做其他事）
> ```java
> int cores = Runtime.getRuntime().availableProcessors();
> int poolSize = cores + 1;  // CPU 密集型
> ```

## 拓展延伸

### synchronized vs ReentrantLock

| 对比 | synchronized | ReentrantLock |
|------|-------------|---------------|
| 使用方式 | 自动加解锁 | 手动 lock/unlock |
| 可中断 | ❌ | ✅ `lockInterruptibly()` |
| 超时 | ❌ | ✅ `tryLock(5, TimeUnit.SECONDS)` |
| 公平锁 | 非公平 | 可设公平 |
| 条件 | 单个 wait/notify | 多个 Condition |
| 性能 | JDK6 优化后差异不大 | 差异不大 |
