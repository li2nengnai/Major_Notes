## 一、JVM 内存结构（运行时数据区）

```
┌─────────────────────────────────────────────────┐
│                    JVM 内存                       │
├─────────────────────┬───────────────────────────┤
│  线程私有            │    线程共享                 │
├─────────────────────┼───────────────────────────┤
│  程序计数器（PC）    │  堆（Heap）                │
│  虚拟机栈（VM Stack）│  方法区（Method Area）      │
│  本地方法栈          │  运行时常量池              │
└─────────────────────┴───────────────────────────┘
```

### 1.1 程序计数器（PC Register）
- 当前线程执行字节码的行号指示器
- 线程私有，无 OOM
- 唯一不会 OOM 的区域

### 1.2 虚拟机栈（VM Stack）
- 每个方法调用创建一个栈帧
- 栈帧：局部变量表 + 操作数栈 + 动态链接 + 方法出口
- **StackOverflowError**：栈深度超限
- **OutOfMemoryError**：动态扩展时无法申请到内存

### 1.3 堆（Heap）⭐
- **所有线程共享**，存放对象实例
- Java 垃圾回收的主要区域（GC 堆）
- 分代：年轻代（Eden/S0/S1）+ 老年代

```
Heap
├── Young Generation（年轻代）
│   ├── Eden（8/10）
│   ├── Survivor 0（S0，1/10）
│   └── Survivor 1（S1，1/10）
│   （比例默认 8:1:1）
└── Old Generation（老年代）
```

### 1.4 方法区（Method Area）
- 存储类信息、常量、静态变量
- JDK 8：**元空间（Metaspace）**，使用本地内存
- JDK 7 及之前：永久代（PermGen）
- 字符串常量池 JDK 7 移到堆

### 1.5 本地方法栈（Native Method Stack）
- 执行 native 方法
- HotSpot 中与虚拟机栈合并

## 二、对象创建过程

```
类加载检查 → 分配内存 → 初始化零值 → 设置对象头 → 执行构造方法
```

**分配方式**：
- 指针碰撞（规整内存）→ Serial、ParNew
- 空闲列表（不规整内存）→ CMS

**并发安全**：
- CAS + 失败重试
- TLAB（Thread Local Allocation Buffer），线程本地分配

**对象内存布局**：
```
┌──────────────────────────────┐
│  对象头                      │
│  ├── Mark Word（标记字段）    │
│  └── Klass Pointer（类指针）  │
├──────────────────────────────┤
│  实例数据（Instance Data）    │
├──────────────────────────────┤
│  对齐填充（Padding）          │
└──────────────────────────────┘
```

## 三、类加载机制

### 3.1 加载过程
```
加载 → 验证 → 准备 → 解析 → 初始化 → 使用 → 卸载
（前5步统称「类加载」）
```

### 3.2 双亲委派模型

```
┌─────────────────────┐
│ Bootstrap ClassLoader │  C++实现，加载 rt.jar
└────────┬────────────┘
         ↓
┌─────────────────────┐
│ Extension ClassLoader│  加载 ext 目录
└────────┬────────────┘
         ↓
┌─────────────────────┐
│ Application ClassLoader│ 加载 classpath
└────────┬────────────┘
         ↓
┌─────────────────────┐
│  自定义 ClassLoader    │
└─────────────────────┘
```

**工作原理**：加载一个类时，先委托父加载器加载，父加载器无法加载时才自己加载。

**好处**：
- 避免重复加载
- 核心 API 不被篡改（如自己写的 `java.lang.String` 不会被加载）

**破坏双亲委派**：SPI（ServiceLoader）、热部署、Tomcat

## 四、垃圾回收（GC）

### 4.1 判断对象是否存活
- **引用计数法**：循环引用问题（Java 不用）
- **可达性分析算法**：从 GC Roots 出发，不可达即死亡

**GC Roots**：
- 虚拟机栈引用对象
- 方法区静态/常量引用对象
- 本地方法栈 JNI 引用

### 4.2 四种引用类型
| 类型 | 回收时机 | 用途 |
|------|---------|------|
| 强引用 | 永不回收 | `new Object()` |
| 软引用 | OOM 前回收 | 缓存 |
| 弱引用 | 下次 GC 回收 | ThreadLocal |
| 虚引用 | 随时 | 管理直接内存 |

### 4.3 GC 算法

| 算法 | 原理 | 优点 | 缺点 |
|------|------|------|------|
| **标记-清除** | 标记存活→清除死亡 | 简单 | 内存碎片 |
| **复制** | 等分两块，存活复制 | 无碎片 | 占用双倍空间 |
| **标记-整理** | 标记存活→向前移动 | 无碎片 | 效率较低 |
| **分代收集** | 新生代复制，老年代标记整理 | 综合最优 | — |

### 4.4 垃圾收集器

```
新生代                    老年代
Serial（单线程+复制）      Serial Old（单线程+标记整理）
ParNew（多线程+复制）       CMS（并发+标记清除）
Parallel Scavenge（吞吐量优先） Parallel Old（标记整理）
                 G1（JDK 9+默认，分区+复制+标记整理）
                 ZGC（JDK 15+，低延迟）
```

**G1**（Garbage First）：
- 将堆分成 2048 个 Region
- 维护优先列表，优先回收垃圾最多的 Region
- 目标：可预测的停顿时间

## 五、JVM 调优常用参数

### 堆设置
```bash
-Xms4g              # 初始堆大小
-Xmx4g              # 最大堆大小（建议和-Xms一致）
-Xmn2g              # 年轻代大小
-XX:MetaspaceSize=256m  # 元空间大小
-XX:+UseG1GC        # 使用 G1 收集器
```

### 日志
```bash
-XX:+PrintGCDetails         # 打印 GC 详情
-XX:+PrintGCDateStamps      # 打印 GC 时间戳
-Xloggc:/path/gc.log        # GC 日志路径
```

### OOM 排查询
```bash
-XX:+HeapDumpOnOutOfMemoryError    # OOM 时 dump 堆
-XX:HeapDumpPath=/path/dump.hprof  # dump 路径
```
